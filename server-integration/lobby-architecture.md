# CS2 内战平台大厅功能架构与思路规划

本文档记录了关于为该项目添加“多房间大厅”功能的初步想法、可行性分析以及最终确定的落地方案。

## 1. 初始构想
* **需求目标**：在搭建好本地 CS2 测试服务器的基础上，为 Web 项目添加一个“大厅”功能。
* **功能描述**：大厅列表中能够显示当前所有的对战房间，并实时展示每个房间的具体信息，包括：
  * 当前游戏状态（例如：等待中、抽卡中、游戏中）
  * 房间在线人数（例如：8/10）
  * 当前游玩的模式或规则集

## 2. 技术可行性分析 (完全可行)
实现上述功能在技术上是非常标准的“对战平台”架构，主要分为数据来源的两个层面：

1. **业务状态 (Web层)**：
   * 房间的“抽卡模式”、“等待状态”等信息，本身就由 Node.js 后端进行管理。
   * 这部分数据直接保存在 Node.js 的内存（或 Redis 等缓存）中，并通过 WebSocket 下发给大厅前端。
2. **游戏真实状态 (Game层)**：
   * 真正的“当前在线人数”、“所在地图”等信息，需要向 CS2 游戏服务端索取。
   * **核心技术点**：Node.js 可以使用 V社官方提供的 **A2S_INFO (Server Query Protocol)**，通过发送极小的 UDP 数据包，瞬间获取 CS2 服务端的状态（无需 RCON 密码），从而展示到网页的大厅列表中。

## 3. 用户体系方案选型
引入多房间和大厅后，必须解决“如何区分玩家身份”的问题。我们探讨了三种方案：

* **方案一：“开黑房”模式（派对游戏模式）**
  * **机制**：无需注册。玩家打开网页输入一个**临时昵称**，系统分配一个 Socket Session ID。通过输入**4位数房间码**加入房间。
  * **优点**：极度轻量，零门槛，适合朋友开黑快速测试。
* **方案二：接入 Steam 官方登录（专业平台模式）**
  * **机制**：通过 OpenID 接入 Steam，一键登录获取 SteamID 和头像。
  * **优点**：最专业，便于未来做战绩统计和与服务端插件深度绑定。
* **方案三：传统账号密码注册**
  * **机制**：填写账号、密码、手机号。
  * **缺点**：门槛极高，开发繁琐，极度劝退开黑玩家。

## 4. 最终确定方案
**决策：采用【方案一】作为初期的实现目标。**

这种类似《胡闹厨房》或 Jackbox 的轻量级派对游戏模式，能以最小的开发成本快速验证核心玩法。

### 未来开发计划拆解
当决定开始开发时，我们将按照以下步骤进行改造：

#### 后端改造 (Node.js)
1. **引入 RoomManager**：在内存中管理多个房间的生命周期（创建、销毁）。
2. **扩展 Socket 事件**：
   * 增加玩家设置昵称的逻辑。
   * 提供 `create_room` (生成 4 位房间码) 和 `join_room` (根据房间码加入) 接口。
   * 提供 `get_lobby_info` 接口，定期推送全站活跃房间列表。
3. **集成 Server Query**：引入 `gamedig` 等库，将分配给特定房间的 CS2 测试服状态整合进大厅数据中。

#### 前端改造 (Vue 3)
1. **入口页 (Login/Name)**：一个极简页面，仅包含一个昵称输入框。
2. **大厅页 (Lobby)**：
   * 左侧：渲染活跃房间列表卡片（显示人数、状态、房间名）。
   * 右侧：“创建房间”按钮与“输入房间码”加入框。
3. **房间页 (Room)**：即现有的核心抽卡和内战控制界面，需增加显示当前房间号。
